# I18n Audit

Date: 2026-03-13

Scope:
- frontend code under `components/`, `pages/`, `hooks/`, `services/`, `utils/`
- locale files under `public/locales/{cs,en,de,pl,sk,at}/translation.json`

Method:
- static scan of `t('...')` key usage
- comparison of used keys against locale JSONs
- spot-check of high-risk screens with missing keys or hardcoded copy

## Summary

The localization layer is functional, but there are several incomplete areas:

- key/schema drift between code and locale JSON
- new UI namespaces added in code but never added to locales
- Czech hardcoded copy still embedded in interactive flows
- a large number of fallbacks rely on `defaultValue`, which prevents breakage but leaves non-Czech locales partially untranslated

Approximate count of used keys missing from locale files:

- `cs`: 472
- `en`: 511
- `de`: 491
- `pl`: 557
- `sk`: 556
- `at`: 557

Note:
- many of these do not surface as raw broken UI because the code often supplies `defaultValue`
- the highest-priority issues are the ones without `defaultValue` or the ones affecting SEO/high-traffic flows

## Priorities

### P0

1. Fix SEO key mismatches in `utils/seo.ts`
   Files:
   - `utils/seo.ts`
   - `public/locales/*/translation.json`

   Problem:
   - code references keys that do not exist under those exact paths:
   - `marketplace.detail`
   - `seo.marketplace_title`
   - `welcome.faq.q1`
   - `welcome.faq.a2`
   - `welcome.faq.q4`
   - `welcome.faq.a5`

   Impact:
   - broken or low-quality page titles/descriptions
   - invalid or poor FAQ structured data across all locales

   Recommended fix:
   - either update `utils/seo.ts` to use the current locale schema
   - or add compatibility keys to all locale files
   - prefer normalizing on one canonical schema and removing legacy key paths

2. Bring `AdminDashboard` onto one translation schema
   Files:
   - `pages/AdminDashboard.tsx`
   - `public/locales/*/translation.json`

   Problem:
   - code mixes flat keys like `admin_dashboard.refresh`
   - some locale files now use nested groups like:
   - `admin_dashboard.header.*`
   - `admin_dashboard.actions.*`
   - `admin_dashboard.common.*`

   Impact:
   - untranslated labels in admin UI
   - some locales are materially behind others
   - future additions are error-prone because the page and JSON structure disagree

   Recommended fix:
   - choose one shape for `admin_dashboard`
   - update page code to that shape
   - then backfill missing keys in all 6 locales

### P1

3. Localize hardcoded candidate journey option text
   Files:
   - `components/AssessmentJourneyFlow.tsx`

   Problem:
   - several button/choice arrays are hardcoded directly in Czech
   - examples include strategy, signals, resource labels, and sequencing options

   Impact:
   - major candidate flow is only partially translated
   - even in English/German sessions the UI can show Czech choices

   Recommended fix:
   - move all interactive option strings into locale keys
   - keep the array structure in code, but source labels from `t(...)`
   - review related `defaultValue` strings in the same component while touching it

4. Complete `invitation_landing.*` for `pl`, `sk`, and `at`
   Files:
   - `pages/InvitationLanding.tsx`
   - `public/locales/pl/translation.json`
   - `public/locales/sk/translation.json`
   - `public/locales/at/translation.json`

   Problem:
   - the page uses many `defaultValue` fallbacks
   - several keys exist in `cs/en/de` but are absent in `pl/sk/at`

   Impact:
   - these locales silently fall back to Czech copy on a public-facing entry page

   Recommended fix:
   - backfill the whole `invitation_landing` namespace first
   - verify the same set against `AssessmentJourneyFlow` because those flows are closely connected

5. Add the missing `activation.*` namespace
   Files:
   - `components/CandidateActivationRail.tsx`
   - `public/locales/*/translation.json`

   Problem:
   - `activation.*` keys are used in code, but the namespace does not exist in any locale file

   Impact:
   - candidate activation rail relies entirely on fallback copy
   - translation coverage is effectively zero for that component

   Recommended fix:
   - add the full namespace to all six locale files
   - prefer English source copy if this component is product-language-neutral

### P2

6. Reduce `defaultValue` debt in large form/workspace screens
   Files:
   - `components/CompanyJobEditor.tsx`
   - `components/AssessmentCreator.tsx`
   - `components/ProfileJobManager.tsx`
   - `components/ApplicationMessageCenter.tsx`
   - `components/ApplicationModal.tsx`
   - `components/company/*`

   Problem:
   - these screens have many translation calls with inline `defaultValue`
   - UX does not break, but locale completeness is weak and hard to maintain

   Impact:
   - non-primary locales get uneven quality
   - translators have no single source of truth

   Recommended fix:
   - migrate the biggest namespaces first:
   - `company.job_editor.*`
   - `profile.job_hub.*`
   - `application.messages.*`
   - `apply.*`

7. Audit locale consistency for company/blog/SEO namespaces
   Files:
   - `public/locales/*/translation.json`

   Problem:
   - several namespaces are partially present only in some languages:
   - `company_landing.blog.*`
   - `company_extra.stats.*`
   - `company.dashboard.stats.*`
   - `seo.base_title_simple`
   - `seo.home_description_simple`

   Impact:
   - inconsistent marketing and dashboard copy across languages

   Recommended fix:
   - generate a locale diff per namespace and normalize one namespace at a time

## Recommended Execution Order

1. `utils/seo.ts` plus locale compatibility fix
